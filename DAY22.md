## The problem with events nobody watches

Every day since Day 17 has made the audit log richer. Day 18 added `ACCOUNT_SUSPENDED` and `ACCOUNT_THROTTLED` when the abuse detector trips. Day 20 added `BUDGET_WARNING_ISSUED` and `BUDGET_EXCEEDED_BLOCKED` when a tenant's spend crosses a line. Day 21 added `TENANT_ACCESS_DENIED` when someone probes for another organization's data. Each of these is, by construction, a moment where something in the system decided "this is worth recording."

But recording is all that happens. `_emit()` (Day 18) writes the record and feeds the abuse detector - both of those are *internal* reactions. Nothing tells a *person*. Concretely:

- Acme's analyst gets auto-suspended at 2 AM for tripping `PRIVILEGE_ESC` three times. The `ACCOUNT_SUSPENDED` record sits in `audit_log` with `org_id="acme"`. Acme's on-call engineer finds out at 9 AM when the analyst messages "why can't I log in" - seven hours after the system already knew something was wrong.
- Globex blows through its monthly budget cap on a Saturday. `BUDGET_EXCEEDED_BLOCKED` fires, every subsequent stream for that user gets blocked with zero tokens billed - silently, from the user's perspective, until someone happens to query `GET /billing/usage`.
- Someone runs three `TENANT_ACCESS_DENIED` probes against Acme's org_id from an account that then gets auto-suspended per Day 21's threshold. The suspension is recorded. Acme's security team has no way to know unless they're actively tailing the audit log.
- Day 17's `AUDIT_TAMPER` - the single scariest event type in this entire system, fired only if the hash chain itself doesn't verify - writes one row and stops there.

The gap is the same shape every time: **the audit log is a record of the past, not a signal about the present.** Day 22 closes it by giving a configurable subset of event types a *push* path - Slack, PagerDuty, or a generic webhook - chosen per event type by severity, routed per organization using Day 21's org-aware secrets, and deduplicated so a burst of identical events doesn't become a burst of identical pages.

<a id="what-it-means-here"></a>
## What real-time alerting means here

Real-time alerting in this system means four things:

1. **A rule registry**, `ALERT_RULES`, mapping the `AuditEventType` values that matter to a severity (`info` / `warning` / `critical`) and a set of channels (`slack` / `pagerduty` / `webhook`). Most event types - the majority of `AuditEventType` - have no rule and are never alerted on. This is opt-in, not "alert on everything."
2. **Per-organization channel configuration**, reusing Day 21's `vault.get_secret(secret, org_id)` - Acme's Slack webhook is not Globex's Slack webhook, and a tenant with nothing configured for a channel simply doesn't receive that channel's alerts (no error, no fallback to some shared channel that would itself be a cross-tenant leak).
3. **Dedup**, so that the five `BUDGET_EXCEEDED_BLOCKED` events generated by a user retrying five times in a minute produce one Slack message, not five - while `ACCOUNT_SUSPENDED` and `AUDIT_TAMPER`, which by Day 22's rules should *never* be suppressed, aren't subject to dedup at all.
4. **A dispatcher wired into `_emit()`** (Day 18), so every code path that already produces an audit event - which, after Days 18-21, is every governance decision in the system - automatically gets alert coverage for free, with zero changes to the dozens of call sites that call `AuditLogger` methods.

What this is *not*: a notification preferences UI, an on-call rotation/escalation system (PagerDuty already does that once an alert reaches it), or a replacement for the audit log as the source of truth - `ALERT_DISPATCHED` and `ALERT_SUPPRESSED` are themselves just more audit records, by design.

## Step 22.1 - Alert severity and the rule registry

The registry is the single place that decides "does this event type page anyone, and through what." It is keyed by `AuditEventType`, mirroring how Day 18's `ABUSE_THRESHOLDS` and Day 19's `AGENT_SECRET_SCOPES` are keyed by the enums they govern.

```python
# backend/alerting/alert_schema.py

from dataclasses import dataclass

from backend.audit.audit_schema import AuditEventType


class AlertSeverity(str, Enum):
    INFO = "info"
    WARNING = "warning"
    CRITICAL = "critical"


class AlertChannel(str, Enum):
    SLACK = "slack"
    PAGERDUTY = "pagerduty"
    WEBHOOK = "webhook"


@dataclass(frozen=True)
class AlertRule:
    severity: AlertSeverity
    channels: tuple[AlertChannel, ...]
    dedup_window_seconds: int  # 0 disables dedup - every occurrence alerts
    description: str


@dataclass(frozen=True)
class AlertMessage:
    """The channel-agnostic payload every Notifier receives."""
    org_id: str
    severity: AlertSeverity
    title: str
    body: str
    event_id: str
    payload: dict


# ─── Rules ───────────────────────────────────────────────────────────────
#
# Only event types listed here ever alert. Adding a row here is the entire
# integration step for "make X page someone" - no changes to the code that
# emits X are needed, because _emit() (Step 22.6) checks this registry for
# every event it writes.
#
ALERT_RULES: dict[AuditEventType, AlertRule] = {

    # The single scariest event type in the system - the hash chain itself
    # didn't verify. Every channel, never suppressed.
    AuditEventType.AUDIT_TAMPER: AlertRule(
        severity=AlertSeverity.CRITICAL,
        channels=(AlertChannel.SLACK, AlertChannel.PAGERDUTY, AlertChannel.WEBHOOK),
        dedup_window_seconds=0,
        description="Audit log hash chain failed verification - possible tampering",
    ),

    # Day 18 - an account was auto-suspended. Someone should know now,
    # not at the next standup.
    AuditEventType.ACCOUNT_SUSPENDED: AlertRule(
        severity=AlertSeverity.CRITICAL,
        channels=(AlertChannel.SLACK, AlertChannel.PAGERDUTY),
        dedup_window_seconds=0,
        description="An account was automatically suspended by the abuse detector",
    ),

    # Day 21 - cross-tenant probing severe enough to suspend.
    AuditEventType.TENANT_ACCESS_DENIED: AlertRule(
        severity=AlertSeverity.CRITICAL,
        channels=(AlertChannel.SLACK, AlertChannel.PAGERDUTY),
        dedup_window_seconds=1800,
        description="Cross-tenant access was denied - possible org_id enumeration",
    ),

    # Day 19 - an agent's outbound message contained a known secret value
    # and was blocked before it left the system.
    AuditEventType.SECRET_LEAK_BLOCKED: AlertRule(
        severity=AlertSeverity.CRITICAL,
        channels=(AlertChannel.SLACK, AlertChannel.PAGERDUTY),
        dedup_window_seconds=0,
        description="A secret value was detected in an outbound message and blocked",
    ),

    # Day 20 - a user is now blocked from further LLM calls until the
    # next period. Worth knowing, not worth paging.
    AuditEventType.BUDGET_EXCEEDED_BLOCKED: AlertRule(
        severity=AlertSeverity.WARNING,
        channels=(AlertChannel.SLACK,),
        dedup_window_seconds=3600,
        description="A user's daily or monthly budget cap was exceeded; streaming blocked",
    ),

    # Day 20 - approaching the cap. Purely informational.
    AuditEventType.BUDGET_WARNING_ISSUED: AlertRule(
        severity=AlertSeverity.INFO,
        channels=(AlertChannel.SLACK,),
        dedup_window_seconds=3600,
        description="A user crossed the warning threshold for their budget period",
    ),

    # Day 18 - an account was throttled (not suspended). Lower severity
    # than ACCOUNT_SUSPENDED, longer dedup window.
    AuditEventType.ACCOUNT_THROTTLED: AlertRule(
        severity=AlertSeverity.WARNING,
        channels=(AlertChannel.SLACK,),
        dedup_window_seconds=1800,
        description="An account was throttled by the abuse detector",
    ),
}


def get_alert_rule(event_type: AuditEventType) -> AlertRule | None:
    return ALERT_RULES.get(event_type)
```

(`from enum import Enum` belongs at the top of the file alongside the `dataclass` import - omitted above only for brevity in the diff.)

Six rules out of the dozens of `AuditEventType` members defined across Days 17-21. That ratio is the point: `GOAL_BLOCKED`, `MESSAGE_REDACTED`, `HITL_CREATED`, and most others remain audit-only. If a future day decides one of those should page someone, it's a one-entry addition here - nothing else changes.

Day 21's `vault.get_secret(secret, org_id)` already checks `<ORG>_<BASE_VAR>` before `<BASE_VAR>` - so `ACME_SLACK_WEBHOOK_URL` is checked before `SLACK_WEBHOOK_URL` for an Acme alert. No vault changes are needed; Day 22 only adds the three new `SecretName` members the alerting code will ask for.

One thing these three secrets deliberately do *not* get is an entry in `AGENT_SECRET_SCOPES` (Day 19). That registry governs which *agents* (`research`, `code_review`) can read which secrets via `CredentialBroker.get_secret()` - it has nothing to do with the alerting pipeline, which is a system component, not an agent, and calls `vault.get_secret()` directly. Keeping alert-channel secrets out of `AGENT_SECRET_SCOPES` means no agent can ever read - or have `redact_secrets()` (Day 19) accidentally treat as sensitive output - your Slack webhook URL.

## Step 22.3 - Channel notifiers: Slack, PagerDuty, webhook

Each notifier resolves its own secret (org-aware), does nothing if unconfigured, and returns whether delivery succeeded. None of them raise on a missing secret - "this org hasn't set up Slack" is a normal, silent no-op, not an error.

```python
# backend/alerting/notifiers.py

from __future__ import annotations

import logging
from typing import Protocol

import httpx

from backend.alerting.alert_schema import AlertMessage, AlertSeverity, AlertChannel
from backend.secrets import vault
from backend.secrets.secret_schema import SecretName

logger = logging.getLogger(__name__)

HTTP_TIMEOUT_SECONDS = 5.0


class Notifier(Protocol):
    async def send(self, message: AlertMessage) -> bool:
        """Return True if the alert was accepted by the channel."""
        ...


class SlackNotifier:
    _EMOJI = {
        AlertSeverity.INFO: ":information_source:",
        AlertSeverity.WARNING: ":warning:",
        AlertSeverity.CRITICAL: ":rotating_light:",
    }

    async def send(self, message: AlertMessage) -> bool:
        webhook_url = vault.get_secret(SecretName.SLACK_WEBHOOK_URL, message.org_id)
        if not webhook_url:
            logger.info("Slack alert skipped for org '%s' - no webhook configured", message.org_id)
            return False

        text = f"{self._EMOJI[message.severity]} *{message.title}*\n{message.body}"
        async with httpx.AsyncClient(timeout=HTTP_TIMEOUT_SECONDS) as client:
            resp = await client.post(webhook_url, json={"text": text})
        return resp.status_code < 300


class PagerDutyNotifier:
    _EVENTS_API_URL = "https://events.pagerduty.com/v2/enqueue"

    async def send(self, message: AlertMessage) -> bool:
        # PagerDuty pages a human on call - reserve it for CRITICAL even if
        # a rule lists PAGERDUTY among its channels for lower severities.
        if message.severity != AlertSeverity.CRITICAL:
            return False

        routing_key = vault.get_secret(SecretName.PAGERDUTY_ROUTING_KEY, message.org_id)
        if not routing_key:
            logger.info("PagerDuty alert skipped for org '%s' - no routing key configured", message.org_id)
            return False

        payload = {
            "routing_key": routing_key,
            "event_action": "trigger",
            "dedup_key": message.event_id,
            "payload": {
                "summary": message.title,
                "severity": "critical",
                "source": f"ai-governance:{message.org_id}",
                "custom_details": message.payload,
            },
        }
        async with httpx.AsyncClient(timeout=HTTP_TIMEOUT_SECONDS) as client:
            resp = await client.post(self._EVENTS_API_URL, json=payload)
        return resp.status_code < 300


class WebhookNotifier:
    async def send(self, message: AlertMessage) -> bool:
        url = vault.get_secret(SecretName.ALERT_WEBHOOK_URL, message.org_id)
        if not url:
            return False

        async with httpx.AsyncClient(timeout=HTTP_TIMEOUT_SECONDS) as client:
            resp = await client.post(url, json={
                "org_id": message.org_id,
                "severity": message.severity.value,
                "title": message.title,
                "body": message.body,
                "event_id": message.event_id,
                "payload": message.payload,
            })
        return resp.status_code < 300


NOTIFIERS: dict[AlertChannel, Notifier] = {
    AlertChannel.SLACK: SlackNotifier(),
    AlertChannel.PAGERDUTY: PagerDutyNotifier(),
    AlertChannel.WEBHOOK: WebhookNotifier(),
}
```

The `dedup_key` PagerDuty receives is the audit record's `event_id` (Day 17) - if the same alert were somehow dispatched twice (it shouldn't be, after Step 22.4, but defense in depth costs nothing here), PagerDuty itself collapses it into one incident rather than opening two.

## Step 22.4 - Dedup and rate limiting

One Redis `SET ... NX EX` per `(org_id, event_type)`, exactly the pattern Day 9 used for `stream:active:...` slots and Day 18 used for signal windows - whoever sets the key first within the window "wins" and everyone else is suppressed until it expires.

```python
# backend/alerting/dedup.py

from backend.cache.redis_client import get_redis


def _dedup_key(org_id: str, event_type: str) -> str:
    return f"alert:dedup:org:{org_id}:{event_type}"


async def should_suppress(org_id: str, event_type: str, window_seconds: int) -> bool:
    """
    True if an alert for this (org, event_type) was already dispatched
    within window_seconds. A window of 0 disables dedup entirely - every
    matching event alerts, used for rules where missing even one
    occurrence (ACCOUNT_SUSPENDED, AUDIT_TAMPER, SECRET_LEAK_BLOCKED) is
    unacceptable.

    Implemented as "first one in wins": the first call within the window
    sets the key and returns False (not suppressed); every subsequent
    call before expiry finds the key already set and returns True.
    """
    if window_seconds <= 0:
        return False

    r = await get_redis()
    accepted = await r.set(_dedup_key(org_id, event_type), "1", nx=True, ex=window_seconds)
    return not accepted
```

`r.set(..., nx=True, ex=window_seconds)` returns a truthy value only if the key didn't already exist (and was therefore just created by this call). `should_suppress` inverts that: the call that *creates* the key is the one that gets to alert; everything after it, until the key expires, is suppressed.

## Step 22.5 - The alert dispatcher

`dispatch_alert()` takes the `AuditLogEntry` that `AuditWriter.append()` (Day 17, updated Day 21) just returned - it already has `org_id`, `event_type`, `event_id`, `actor_id`, `outcome`, `reason`, and `payload` (as a JSON string). The dispatcher decodes the payload, checks the rule and dedup, fans out to channels, and writes its own outcome record.

```python
# backend/alerting/dispatcher.py

import json
import logging

from backend.audit.audit_schema import AuditEvent, AuditEventType
from backend.database.models import AuditLogEntry
from backend.alerting.alert_schema import get_alert_rule, AlertMessage
from backend.alerting.notifiers import NOTIFIERS
from backend.alerting.dedup import should_suppress

logger = logging.getLogger(__name__)


def _format_body(entry: AuditLogEntry, payload: dict) -> str:
    lines = [
        f"org: {entry.org_id}",
        f"actor: {entry.actor_id} ({entry.actor_role or 'unknown'})",
        f"outcome: {entry.outcome}",
    ]
    if entry.reason:
        lines.append(f"reason: {entry.reason}")
    if payload:
        lines.append(f"details: {json.dumps(payload, default=str)}")
    return "\n".join(lines)


async def dispatch_alert(entry: AuditLogEntry, writer) -> None:
    """
    Called from _emit() (Step 22.6) after an audit record is written.

    Never raises: alerting is best-effort and must not affect the
    governance decision that produced the record it's reacting to.
    Writes its own ALERT_DISPATCHED / ALERT_SUPPRESSED / ALERT_DELIVERY_FAILED
    record via writer.write() directly (not _emit()) - none of those
    three event types has an ALERT_RULES entry, so there is no risk of
    recursion, and they aren't behavioral signals the abuse detector
    needs to see.
    """
    try:
        event_type = AuditEventType(entry.event_type)
    except ValueError:
        return  # unknown/legacy event_type string - nothing to do

    rule = get_alert_rule(event_type)
    if rule is None:
        return  # not an alertable event type

    try:
        payload = json.loads(entry.payload) if entry.payload else {}
    except (TypeError, ValueError):
        payload = {}

    if await should_suppress(entry.org_id, entry.event_type, rule.dedup_window_seconds):
        writer.write(AuditEvent(
            event_type=AuditEventType.ALERT_SUPPRESSED,
            org_id=entry.org_id, actor_id="system", outcome="suppressed",
            reason=f"duplicate {entry.event_type} within {rule.dedup_window_seconds}s",
            payload={"source_event_id": entry.event_id},
        ))
        return

    message = AlertMessage(
        org_id=entry.org_id,
        severity=rule.severity,
        title=f"[{rule.severity.value.upper()}] {entry.event_type}",
        body=_format_body(entry, payload),
        event_id=entry.event_id,
        payload=payload,
    )

    delivered: list[str] = []
    for channel in rule.channels:
        try:
            if await NOTIFIERS[channel].send(message):
                delivered.append(channel.value)
        except Exception:
            logger.exception(
                "Alert delivery failed: org=%s channel=%s event=%s",
                entry.org_id, channel.value, entry.event_type,
            )
            writer.write(AuditEvent(
                event_type=AuditEventType.ALERT_DELIVERY_FAILED,
                org_id=entry.org_id, actor_id="system", outcome="failed",
                reason=f"{channel.value} delivery raised an exception",
                payload={"source_event_id": entry.event_id, "channel": channel.value},
            ))

    writer.write(AuditEvent(
        event_type=AuditEventType.ALERT_DISPATCHED,
        org_id=entry.org_id, actor_id="system",
        outcome="dispatched" if delivered else "no_channel_configured",
        reason=rule.description,
        payload={
            "source_event_id": entry.event_id,
            "channels_delivered": delivered,
            "channels_configured": [c.value for c in rule.channels],
            "severity": rule.severity.value,
        },
    ))
```

`outcome="no_channel_configured"` (when `delivered` is empty but a rule matched and dedup didn't suppress) is itself useful signal - it means an org has a `CRITICAL` rule like `ACCOUNT_SUSPENDED` but no Slack or PagerDuty secret configured at all. `GET /alerts/recent` (Step 22.8) surfaces this so an admin can notice "we have no alerting configured" before the first real incident, not during it.

## Step 22.6 - Wiring dispatch into `_emit()`

This is the one-function change that gives every existing `AuditLogger` call site - all of Days 17 through 21 - alert coverage. `_emit()` (Day 18) already does two things after `_writer.write(event)`: it's the only place that touches every audit event uniformly. Dispatch becomes the third thing.

```python
# backend/audit/audit_logger.py (updated _emit, building on Day 18 + Day 21)

import asyncio

from backend.alerting.dispatcher import dispatch_alert


def _emit(event: AuditEvent) -> AuditLogEntry:
    entry = _writer.write(event)

    # Day 18: abuse detection (now org-scoped per Day 21's namespaced
    # signal keys - this block is otherwise unchanged from Day 18/21)
    decision = _detector.evaluate(event)
    if decision is not None:
        _writer.write(AuditEvent(
            event_type=AuditEventType.ABUSE_DETECTED,
            org_id=event.org_id, actor_id=event.actor_id, actor_role=event.actor_role,
            session_id=event.session_id, outcome=decision.action.value, reason=decision.reason,
            payload={
                "signal_type": decision.signal_type.value,
                "count": decision.count,
                "threshold": decision.threshold,
                "window_seconds": decision.window_seconds,
            },
        ))
        if decision.action == AbuseAction.SUSPEND:
            suspensions.suspend(event.org_id, event.actor_id, decision.reason, SUSPEND_DURATION_SECONDS)
            _writer.write(AuditEvent(
                event_type=AuditEventType.ACCOUNT_SUSPENDED,
                org_id=event.org_id, actor_id=event.actor_id, actor_role=event.actor_role,
                session_id=event.session_id, outcome="suspended", reason=decision.reason,
                payload={"duration_seconds": SUSPEND_DURATION_SECONDS},
            ))
        elif decision.action == AbuseAction.THROTTLE:
            suspensions.throttle(event.org_id, event.actor_id, THROTTLE_DURATION_SECONDS, THROTTLE_FACTOR)
            _writer.write(AuditEvent(
                event_type=AuditEventType.ACCOUNT_THROTTLED,
                org_id=event.org_id, actor_id=event.actor_id, actor_role=event.actor_role,
                session_id=event.session_id, outcome="throttled", reason=decision.reason,
                payload={"duration_seconds": THROTTLE_DURATION_SECONDS, "factor": THROTTLE_FACTOR},
            ))

    # Day 22: real-time alerting - fire and forget
    _schedule_alert_dispatch(entry)

    return entry


def _schedule_alert_dispatch(entry: AuditLogEntry) -> None:
    """
    dispatch_alert() is async (it makes HTTP calls); _emit() is sync and
    is called from both async request handlers and plain sync test code.
    Handle both:
      - inside a running event loop (normal FastAPI request path):
        schedule it as a background task and return immediately - the
        caller's response is never delayed by Slack/PagerDuty latency.
      - no running loop (a script or a sync test calling _emit() directly):
        run it to completion with asyncio.run().
    Either way, exceptions from dispatch_alert() never propagate here -
    dispatch_alert() already catches per-channel errors, and a failure
    to even schedule it is logged and swallowed.
    """
    try:
        loop = asyncio.get_running_loop()
        loop.create_task(dispatch_alert(entry, _writer))
    except RuntimeError:
        try:
            asyncio.run(dispatch_alert(entry, _writer))
        except Exception:
            logger.exception("Synchronous alert dispatch failed for event %s", entry.event_id)
```

Two small details worth calling out. First, `suspensions.suspend()` and `suspensions.throttle()` (Day 18) now take `org_id` as their first argument - a one-line consequence of Day 21's namespacing that this listing makes explicit since it's the only place those functions are called. Second, "fire and forget" really does mean the HTTP request that triggered, say, `BUDGET_EXCEEDED_BLOCKED` returns to its caller without waiting on Slack - if Slack is slow or down, the user still gets their `429`/blocked-stream response promptly, and `ALERT_DELIVERY_FAILED` (Step 22.7) records the channel failure separately.

## Step 22.7 - New audit events for the alert pipeline itself

```python
# backend/audit/audit_schema.py (addition to AuditEventType)

    # Alerting (new)
    ALERT_DISPATCHED      = "alert.dispatched"
    ALERT_SUPPRESSED      = "alert.suppressed"
    ALERT_DELIVERY_FAILED = "alert.delivery_failed"
```

These three are written by `dispatch_alert()` via `writer.write()` directly, as shown in Step 22.5 - never via `_emit()`, and never present as keys in `ALERT_RULES`. Both choices are the same guard against the same failure mode: if `ALERT_DISPATCHED` itself routed back through `_emit()` and had a rule, every alert would generate another alert about itself, forever. Day 18 already established the precedent for "administrative records that bypass `_emit()`" with `session_blocked_suspended()` and `suspension_lifted()`; `ALERT_DISPATCHED`/`ALERT_SUPPRESSED`/`ALERT_DELIVERY_FAILED` are the same category.

from fastapi import APIRouter, Depends, HTTPException

    if ctx.user_role != "admin":

            "channels": [c.value for c in rule.channels],
            "dedup_window_seconds": rule.dedup_window_seconds,

        }
        for event_type, rule in ALERT_RULES.items()
    ]

@router.get("/alerts/channel-status")

    _require_admin(ctx)

    return {
        "org_id": ctx.org_id,
        "slack_configured": vault.is_configured(SecretName.SLACK_WEBHOOK_URL, ctx.org_id),
        "pagerduty_configured": vault.is_configured(SecretName.PAGERDUTY_ROUTING_KEY, ctx.org_id),
        "webhook_configured": vault.is_configured(SecretName.ALERT_WEBHOOK_URL, ctx.org_id),
    }

## What you built today

An `ALERT_RULES` registry mapping a deliberately small set of `AuditEventType` values - `AUDIT_TAMPER`, `ACCOUNT_SUSPENDED`, `ACCOUNT_THROTTLED`, `TENANT_ACCESS_DENIED`, `SECRET_LEAK_BLOCKED`, `BUDGET_EXCEEDED_BLOCKED`, `BUDGET_WARNING_ISSUED` - to a severity and a tuple of channels. Three new `SecretName` entries for Slack, PagerDuty, and a generic webhook, resolved per-organization through Day 21's existing `vault.get_secret(secret, org_id)` with no vault changes needed. Three `Notifier` implementations - `SlackNotifier`, `PagerDutyNotifier` (which only ever pages for `CRITICAL`), and `WebhookNotifier` - each a silent no-op when its org hasn't configured that channel. A Redis-backed `should_suppress()` that collapses repeated alerts within a rule's window, with `dedup_window_seconds=0` as an explicit "never suppress" for the events where missing one matters most. An async `dispatch_alert()` that looks up the rule, checks dedup, fans out to channels, and writes its own outcome record - and a one-function change to `_emit()` (Day 18) that schedules it as a fire-and-forget background task after every audit write, giving every one of the dozens of `AuditLogger` call sites built since Day 17 alert coverage automatically. Three new audit event types - `ALERT_DISPATCHED`, `ALERT_SUPPRESSED`, `ALERT_DELIVERY_FAILED` - written outside `_emit()` so the alert pipeline can never alert on itself. And three admin endpoints, including `/alerts/channel-status`, so an org can check *before* an incident whether its `CRITICAL` rules actually go anywhere.
